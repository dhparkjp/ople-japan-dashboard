# 오플 재팬 — 고객 인텔리전스 & 멀티채널 통합 대시보드 (데모)

대표 설득용 **개념증명 데모**입니다. "데이터로 운영하면 매출이 이렇게 움직인다"를
세 화면으로 보여줍니다. 실제 운영 시스템·결제·로그인·DB는 없습니다(가상 데이터).

## 실행 방법 (설치 불필요)

Node·npm 없이 **Python만으로** 돕니다.

```bash
cd ople-japan-dashboard
python3 -m http.server 8000
```

그다음 브라우저에서 **http://localhost:8000** 을 엽니다.

> `file://` 로 직접 열면 데이터(JSON) 로드가 막힙니다. 반드시 위 서버로 여세요.

## 데이터 다시 만들기 (선택)

분포를 바꾸거나 새로 뽑고 싶으면:

```bash
python3 generate_data.py
```

`data/*.json` 이 다시 생성됩니다. 시드(`SEED`)가 고정돼 결과는 재현됩니다.

## 화면 구성

상단 네비로 세 영역을 전환합니다.

- **① 통합 관리** — 전 채널 합산 매출 추이·채널 비중·KPI·채널 비교, 마켓→자사몰 구독 퍼널(가정).
- **② 자사몰 (핵심)** — KPI → RFM 세그먼트 → 코호트 리텐션 → **재구매 엔진(오늘 챙길 고객)** → 상품 구성(번들·장바구니) → **연매출 시뮬레이터(방치 vs 실행)**.
- **③ 마켓플레이스** — 라쿠텐·아마존·큐텐의 채널·SKU·가격 비교. *고객 단위 분석은 불가(데이터 제약).*
- **④ 액션 센터** — 위 분석을 "오늘의 할 일"로. 각 액션이 대상 N명·채널·기대 효과 카드로 뜨고, 버튼은 데모 시뮬레이션(실제 발송 없음). "그래서 뭘 하나"의 답.

각 분석 옆 **"ⓘ 이게 왜 매출인가"** 토글에 쉬운 설명을 달았습니다.

## 핵심 데이터 제약 (의도된 설계)

- **자사몰** = 고객 한 명 한 명의 이력 보유 → RFM·재구매 예측 같은 풀 CRM 가능.
- **마켓플레이스** = 채널·상품 단위만 제공 → 고객 추적 불가. 그래서 자사몰이 분석의 본진.

이 차이를 데이터 구조로 그대로 반영했습니다(마켓 주문은 `customer_id` 가 없음).

## 폴더 구조

```
catalog.py        상품 카탈로그(용량·1일섭취·소진일수·환율) — 단일 진실 원천
generate_data.py  가상 데이터 생성 → data/*.json
data/             products·customers·orders·subscriptions·meta
vendor/           echarts.min.js (오프라인 동봉)
index.html
css/styles.css
js/
  data.js         ★ 데이터 접근부 1곳 — 실데이터 전환 시 여기만 교체
  analytics.js    ★ RFM·코호트·소진주기·1→2전환·장바구니·LTV·시뮬레이터 산식
  format.js       엔/원 병기·숫자·날짜 포맷
  charts.js       ECharts 공통 래퍼
  main.js         부트스트랩 + 네비
  views/          integrated · ownstore · marketplace
```

## 실데이터 전환 시 (메모)

- 가장 먼저 확인: 일본 자사몰·각 마켓플레이스에서 실제로 export/연동 가능한 데이터.
- 일본 현실: 리마인더는 LINE, 결제는 콘비니·대금상환 비중↑, 건기식 문구는 薬機法 제약,
  직구는 세제개편 영향, 마켓 ToS상 고객 외부 유인 제한.
- 코드는 `js/data.js` 한 곳만 실제 API로 바꾸면 화면·분석이 그대로 동작하도록 분리돼 있습니다.
